# Анализ кода и Excel-данных

## Кратко
- Генератор читает `input_softitek.xlsx` напрямую через XML внутри `.xlsx` (без `openpyxl/pandas`) и формирует два ST-артефакта: `GVL.txt` и `Proc_IO.txt`.
- Текущие данные Excel в целом консистентны: `AI=10`, `RTD=7`, `DI=7`, `DO=3`, `lines=32`, `dt=23`.
- В репозитории лежат `GVL.txt`/`Proc_IO.txt`, которые **не соответствуют** текущему `input_softitek.xlsx` (в них остался старый `count_ai=11`).

## Как устроен код

### 1) Чтение Excel
Файл `st_codegen/xlsx_plc_reader.py`:
- открывает `.xlsx` как ZIP,
- читает `sharedStrings.xml`,
- находит нужный лист по имени через `workbook.xml` + `workbook.xml.rels`,
- парсит строки/ячейки в словари вида `{номер_колонки: значение}`.

Ключевые функции:
- `read_plc_config()` — параметры листа `PLC` (`count ai`, `count rtd`, ...);
- `read_rtd_points()` — список RTD-тегов (`tag`, `module`, `channel`);
- `read_phase_points()` — фазировка по линиям с преобразованием `А/В/С` в код `1..7`.

### 2) Генерация ST
- `st_codegen/gvl_generator.py` — шаблон констант/массивов GVL на основе `PlcConfig`.
- `st_codegen/proc_io_generator.py`:
  - генерирует инициализацию `TOF_AlarmRTD` для RTD-модулей,
  - маппит первые `count_dt + 1` RTD-точек в `GVL.DataPack.aDT[...]`,
  - инициализирует фазировку `GVL.inNumL[...]`,
  - формирует массив ошибок модулей `GVL.Err_mod[...]`.

## Проверка Excel (фактические данные)

Проверка выполнялась скриптом через текущий парсер.

### PLC
- `count_ai = 10`
- `count_rtd = 7`
- `count_di = 7`
- `count_do = 3`
- `count_lines = 32`
- `count_dt = 23`
- `first_group = 17`

### Таблицы сигналов
- AI: 80 валидных строк, 10 модулей, по 8 каналов/модуль, дубликатов нет.
- RTD: 28 валидных строк, 7 модулей, по 4 канала/модуль, дубликатов нет.
- DI: 112 валидных строк, 7 модулей, по 16 каналов/модуль, дубликатов нет.
- DO: 48 валидных строк, 3 модуля, по 16 каналов/модуль, дубликатов нет.
- Phase: 32 записи, покрытие линий 1..32, уникальность линий полная.

### Важный вывод по данным
`count_dt = 23` означает, что в генерации используются 24 датчика (`DT[0]..DT[23]`), а в RTD-таблице их 28 — это допустимо (4 RTD-точки остаются неиспользованными в `aDT`).

## Обнаруженное расхождение в репозитории

Повторная генерация из текущего `input_softitek.xlsx` показывает различия с коммитнутыми файлами:
- в `GVL.txt` сейчас зафиксировано `count_ai=11` и `gc_count_modules=28`,
- по данным Excel и логике генератора должно быть `count_ai=10` и `gc_count_modules=27`.

Из-за этого в `Proc_IO.txt` смещаются адреса RTD-модулей (`A1_13..A1_19` вместо ожидаемых `A1_12..A1_18`) и границы типов в `Err_mod`.
